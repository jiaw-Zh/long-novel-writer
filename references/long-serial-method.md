# 百万/千万字长篇机制

## 长篇不是章节数放大

百万字以上的核心问题不是"写得更长"，而是让故事在长期推进中持续产生目标、代价、误解、升级和回收。默认把作品设计成多层循环：

- 整书主线：最终问题、终局真相、主角最终代价。
- 卷主线：每卷解决一个阶段性问题，同时暴露更深一层规则。
- 篇章单元：10-30 章形成一个中型目标。
- Chunk：3-5 章形成一个小悬念单元。
- 单章：一个明确推进点，一个情绪钩子，一个读者继续读的理由。

## 记忆层

长期连载的记忆系统在 `references/memory-protocol.md` 中完整定义，包含：

- L0–L5 分层摘要与冻结规则（防止覆盖式摘要稀释早期细节）。
- 实体中心档案（角色/地点/道具/组织/力量体系各自建档、按需加载）。
- 正典账本（`canon/facts.jsonl` + `rules.md` + `timeline.md`，只追加不修改，支持机器级冲突检测）。
- 写前组装协议（9 步固定流程）与写后校验闭环（4 项校验 + 10 步落盘顺序）。

**文件布局、字段定义见 `references/file-contract.md`。**

实际作品至少需要维护：

- `metadata.md`、`story-bible.md`、`continuity-issues.md`
- `summaries/global.md`（L5）与各层 L2/L3/L4 摘要目录
- `chapters/volume-NNN/chapter-NNNN.md` + `.brief.md` + `.index.md`
- `entities/{characters,locations,items,organizations,systems}/`
- `canon/facts.jsonl`、`canon/rules.md`、`canon/timeline.md`
- `foreshadowing-ledger.md`

不要把"没有本地向量库"理解为不需要长期记忆。纯文件 + 章节索引 + 实体档案已能支撑千万字级检索；外部 RAG 或知识库可作为第 9 步检索的替代实现。

## 节奏规则

- 每 3-5 章完成一个小闭环：提出问题、推进误解、制造代价、给出阶段结果。
- 每 10-30 章完成一个篇章目标：胜利不能只是胜利，必须带出新账、新敌人或新规则。
- 每卷结尾必须回收一部分伏笔，同时打开更深层问题。
- 爽点不能只靠升级。交替使用：身份落差、信息差、规则利用、反杀、关系变化、资源夺取、真相揭露、代价承担。

## 一致性规则

写正文前检查（属写前组装协议的一部分，详见 memory-protocol）：

- 主角此刻知道什么，不知道什么。
- 角色是否按已有动机行动。
- 本章是否违反世界规则或实体硬约束。
- 道具、伤势、能力、地理位置是否延续。
- 当前冲突是否推进了卷目标。
- 是否重复最近 20 章用过的桥段（对照 `chapter-*.index.md`→`plot_beats`）。

写正文后按 memory-protocol §5 的落盘顺序更新：

- L0/L1/索引优先落盘。
- canon 账本（facts/timeline/rules）在实体档案之前追加。
- 实体档案追加「关键节点」与「变更记录」，不覆盖旧事实。
- 伏笔账本更新状态。
- 到达 chunk/arc/volume 末章时生成对应 L2/L3/L4。

## 检索/RAG 规则

当作品进入百万字规模，默认需要检索层。可用本地文件检索、BM25、向量库、外部 RAG API 或用户提供的知识库。没有向量库时，`chapter-*.index.md` 的 keywords 字段加简单文本检索即可支撑。

检索前生成 3-5 组关键词：

- 实体 + 属性
- 事件 + 后果
- 地点 + 特征
- 角色 + 秘密
- 伏笔 + 回收节点

检索后分类为：情节燃料、人物维度、世界碎片、叙事技法、冲突预警。命中的结果优先取 L1 brief，再按需展开到对应 L0 正文段。
